# x402 compatibility & roadmap

[x402](https://github.com/coinbase/x402) is Coinbase's open, HTTP-native payment protocol
(HTTP `402 Payment Required`) for agents and APIs, live on Base with USDC. It defines three
roles тАФ **client**, **resource server**, and **facilitator** тАФ where the facilitator exposes
**`/verify`** and **`/settle`** (settle submits the payment on-chain and waits for
confirmation), and payment travels in the `PAYMENT-REQUIRED` / `PAYMENT-SIGNATURE` /
`PAYMENT-RESPONSE` headers. Its first scheme is **`exact`** (EIP-3009
`transferWithAuthorization`).

mandatehub does **not** re-implement on-chain settlement тАФ x402 already solves that. Instead
mandatehub is the **mandate + proof layer** that sits inside a facilitator: it decides
*whether an autonomous agent is allowed to make this payment* (budget / policy / session-key /
window / replay), and it *proves* тАФ offline тАФ that every settled payment stayed within the
mandate. That is exactly the piece x402 leaves to the resource server.

## What ships today (`mandatehub.x402`)

A facilitator that mirrors x402's shape, with settlement pluggable:

- **`Facilitator.verify(requirements, payload)`** тЖТ a `VerifyResponse` computed by a
  side-effect-free mandate pre-authorization (`preauthorize`) тАФ the ledger is never touched.
- **`Facilitator.settle(requirements, payload)`** тЖТ a `SettleResponse` that runs the payment
  through the mandate (posting the double-entry transfer) and returns a `ProofOfMandate`.
- **`PaymentRequirements` / `PaymentPayload`** using x402's vocabulary (`scheme`, `network`,
  `maxAmountRequired`, `payTo`, `asset`, `extra`), plus a mandatehub `extra` carrying the
  `mandateId` and `purpose`.
- **`serve_once` + the `PAYMENT-REQUIRED` / `PAYMENT-SIGNATURE` / `PAYMENT-RESPONSE` headers**
  тАФ a stdlib-only HTTP 402 request cycle. [`examples/x402_facilitator.py`](../examples/x402_facilitator.py)
  runs it over a real localhost server: `402 тЖТ pay тЖТ 200 + proof`, and a replayed payment is
  rejected by the mandate over HTTP.

Settlement today uses the self-contained **`LedgerSettlementAdapter`** тАФ it posts to the
mandatehub ledger, **no real money**. It is the mock/testnet substitute for on-chain settle.

## The seam: `SettlementAdapter`

```python
class SettlementAdapter(Protocol):
    def settle(self, engine, requirements, payload, *, at) -> tuple[bool, str, str]:
        """(success, tx_id, reason). Where the payment actually moves value."""
```

Swapping this is the *only* change needed to move real value: an adapter that forwards
`(paymentPayload, paymentRequirements)` to a real x402 facilitator (Coinbase CDP on Base) and
returns the on-chain tx hash тАФ while mandatehub keeps doing the mandate gating + proofs around
it.

## Roadmap тАФ "actually running", modeled on x402

x402 shipped as an open monorepo (a Python SDK under `./python/x402`, `docs/`, `specs/`,
`examples/`) with the `exact` scheme first and more `(scheme, network)` pairs added over time.
mandatehub follows the same shape:

- **Phase 1 тАФ HTTP 402 facilitator (done).** `mandatehub.x402` with the `exact` scheme, the
  header protocol, mandate-gated verify/settle, and a live HTTP example. Settlement is the
  ledger adapter (no real money). Fully tested.
- **Phase 2 тАФ real x402 v1 client (built; tested against stubs).** `mandatehub.x402` now
  ships a `RemoteFacilitatorAdapter` that speaks the real x402 **v1** facilitator protocol
  (`/verify`, `/settle`) over `urllib`, plus an `ExactEvmPayloadBuilder` that constructs the
  `exact`-scheme EIP-3009 + EIP-712 payment payload signed by a pluggable `Signer`. Verified
  against an in-process stub facilitator + `StubSigner` (no network, no keys). Running it
  against **real Base Sepolia** is the operator's last mile (facilitator URL, agent key,
  testnet USDC) тАФ see [┬з Phase 2 details](#phase-2-real-x402-v1-client) below.
- **Phase 3 тАФ best-execution scheme (built; offline accounting layer).** `mandatehub.x402`
  now exposes the solver auction + surplus recapture (тСв) as the **`best-exec`** scheme
  alongside `exact`: the facilitator best-executes within the user's max and rebates the
  price-improvement surplus, with `ProofOfBestExecution` + `ProofOfSurplusRecapture` and
  offline third-party re-verification in the response. Specified in
  [`specs/best-exec.md`](../specs/best-exec.md). The audited on-chain settler is out of core тАФ
  see [┬з Phase 3 details](#phase-3-best-exec-scheme) below.
- **Phase 4 тАФ mainnet hardening.** Durable storage (Postgres via the `LedgerStorage`
  protocol), key management, auth/rate-limiting, monitoring, a security review, signed
  releases, and a PyPI publish. Only then is real-money mainnet operation appropriate.

## Phase 2: real x402 v1 client

`mandatehub.x402` now includes a faithful **x402 v1** client for the `exact` (EVM/EIP-3009)
scheme тАФ the format the live Coinbase CDP facilitator on Base speaks today. Everything below
is verified against an in-process stub facilitator + a `StubSigner`, so the wire format,
codec, request/response wiring, EIP-712 assembly, and every security guard are tested
**without any network or keys**.

Pieces (`mandatehub/x402/`, stdlib-only except the optional signer):

- **`wire.py`** тАФ v1 wire dataclasses (`X402PaymentRequirements`, `X402PaymentPayload`,
  `EIP3009Authorization`, `FacilitatorVerifyResult`, `FacilitatorSettleResult`) with exact
  camelCase / `exclude_none` serialization and standard-padded base64 header codecs
  (`X-PAYMENT` / `X-PAYMENT-RESPONSE`).
- **`eip712.py`** тАФ the `TransferWithAuthorization` typed-data assembly and the Base Sepolia
  constants (chain id `84532`, USDC `0x036CbDтАжCF7e`, EIP-712 domain `{name:"USDC",
  version:"2"}`; the domain separator was recomputed on-chain to a byte-exact match).
- **`signer.py`** тАФ a `Signer` protocol with `NullSigner` / `StubSigner` (stdlib).
- **`exact_evm.py`** тАФ `ExactEvmPayloadBuilder`: fresh 32-byte CSPRNG nonce, `value ==
  maxAmountRequired`, `to == payTo`, `validBefore = now + min(maxTimeoutSeconds, mandate_ttl)`.
- **`remote.py`** тАФ `RemoteFacilitatorAdapter`: `https`-only (localhost exception), refuses
  cross-host redirects on `/verify`+`/settle`, never puts secrets in errors, raises on
  non-2xx / malformed JSON / network error (never treats failure as success), tolerant of
  unknown/`error`-vs-`errorReason` response keys.

Real EVM signing lives in **`mandatehub/signers/eth_account_signer.py`** behind the optional
`[evm]` extra (`pip install 'mandatehub[evm]'` тЖТ `eth-account`); the core never imports it.

### Operator's last mile (env / config)

To point at real Base Sepolia, supply (never hardcode):

| env var | purpose |
| ------- | ------- |
| `MANDATEHUB_FACILITATOR_URL` | facilitator base URL (`https://` only; `/verify`,`/settle` appended) |
| `MANDATEHUB_AGENT_PRIVATE_KEY` | EVM signer key (via keystore/KMS ideally) тАФ consumed only by `EthAccountSigner`, never logged |
| `MANDATEHUB_NETWORK` | network slug (default `base-sepolia`) |
| `MANDATEHUB_CDP_API_KEY_*` | CDP auth, injected via the adapter's `header_hook` (CDP header spec still to confirm) |

Safe Base Sepolia defaults (chain id, USDC address, decimals, EIP-712 name/version) are
built in but overridable, and the builder still reads `asset`/`extra` from the runtime
requirements so other tokens/networks work.

### Live smoke test (confirm v1 against a real facilitator)

`examples/x402_live_smoke.py` builds a real payload with `EthAccountSigner` and calls a real
facilitator's `/verify` (and, opt-in, `/settle`). It is **not** part of the offline suite
(it touches the network). Run it with your own credentials:

```bash
pip install 'mandatehub[evm]'
export MANDATEHUB_FACILITATOR_URL=https://x402.org/facilitator   # or your CDP facilitator
export MANDATEHUB_AGENT_PRIVATE_KEY=0x...                        # Base Sepolia agent key
export MANDATEHUB_PAY_TO=0x...                                   # recipient address
python examples/x402_live_smoke.py                              # /verify only (non-destructive)
# MANDATEHUB_LIVE_SETTLE=1 python examples/x402_live_smoke.py    # also /settle (real testnet value)
```

`isValid=false` with `invalidReason: insufficient_funds` still confirms the facilitator
accepted our v1 wire format (just fund the agent wallet with Base Sepolia USDC). A CDP
facilitator needs auth headers тАФ pass them via `RemoteFacilitatorAdapter(header_hook=...)`.

### Still gated before mainnet

The v1 assumption should be confirmed with one live smoke test against the real CDP
facilitator (behind a `MANDATEHUB_LIVE_TEST=1` flag); the CDP auth-header format is
unconfirmed; and grant-side mitigations (server-derived requirements, atomic nonce
check-and-lock, deliver-only-after-`success:true`, fail-closed on `429`) belong to the
mandate-gate layer that wraps the adapter тАФ see the security section of the internal spec.

## Phase 3: best-exec scheme

`mandatehub.x402` exposes тСв as the **`best-exec`** x402 scheme тАФ full spec in
[`specs/best-exec.md`](../specs/best-exec.md). A resource server charges "up to *X*"
(`maxAmountRequired`); the facilitator runs a mandate-gated solver auction, fills at the best
disclosed cost, and rebates the surplus by an agreed policy тАФ a *value-add* facilitator, not
just a pass-through.

Pieces (`mandatehub/x402/best_exec.py`, stdlib-only), all reusing тСв/тСг unchanged
(`run_auction` / `compute_split` / `settle_via_auction` / `ProofOf*` тАФ no logic duplicated):

- **`BestExecParams`** тАФ the `extra.bestExec` requirement: `objective`, `settler`,
  `splitPolicy`, `rebateTo`, `operatorTo`, `currency`, `mandateRef`, `purpose`.
- **`build_best_exec_payload`** тАФ a *fixed-value* EIP-3009 authorization whose **nonce is a
  commitment over the whole binding** (settler, asset, payTo, rebateTo, operatorTo, split
  policy hash, objective, intent, window, chain). The operator cannot repoint funds or skim
  the rebate without invalidating the signature.
- **`BestExecFacilitator.verify` / `settle`** тАФ the verify matrix (`VALUE_NOT_MAX`,
  `OUTSIDE_WINDOW`, `POLICY_MISMATCH` on **every** bound field, `MANDATE_REF_*`, on-chain
  guards, budget/replay, `NO_WINNING_BID`, `GAS_EXCEEDS_SURPLUS`) and a single balanced
  settlement over the two value planes (INV-9: the budget plane only ever books `maxAmount`).
- **`verify_best_exec_response`** тАФ a third party recomputes the accounting from the response
  alone: re-run the auction, `executedCost` = winner, an **independent** no-worse recompute
  (does *not* trust the proof's self-reported flag), candidates Merkle root, within-limit,
  integer-exact split vs the agreed policy, and the disclosed binding's digest = signed nonce.

`python examples/x402_best_exec.py` runs the whole flow: `verify тЖТ settle тЖТ offline re-verify`,
printing the two value planes and all 15 offline checks.

### The honest boundary

`best-exec` proves the **accounting**, offline тАФ not that funds moved on chain (that is one
online tx read, step 9 of the spec). The offline binding/policy hash is **sha256** (stdlib has
no keccak); the on-chain nonce is **keccak256** over the same preimage (two hash domains).
On-chain atomicity, no-withholding, and nonce enforcement depend on an audited
**`BestExecSettler` contract** тАФ **unbuilt, out of this core, and a hard gate (H1)** before any
real value. `settlementPlane: "in-ledger"` in the response says so plainly. See
[`specs/best-exec.md` ┬з 7](../specs/best-exec.md) for the settler's MUST-requirements.

## Honest status

Phase 1 is a **runnable protocol demo**, not a production payment service. It runs the x402
HTTP flow end-to-end and proves the mandate accounting, but it moves no real money and has no
persistence, auth, key management, or on-chain settlement yet. Phases 2тАУ4 are where "actually
running with real value" is earned тАФ each gated on decisions (network, custody, hosting) that
belong to the operator.
