# x402 compatibility & roadmap

[x402](https://github.com/coinbase/x402) is Coinbase's open, HTTP-native payment protocol
(HTTP `402 Payment Required`) for agents and APIs, live on Base with USDC. It defines three
roles — **client**, **resource server**, and **facilitator** — where the facilitator exposes
**`/verify`** and **`/settle`** (settle submits the payment on-chain and waits for
confirmation), and payment travels in the `PAYMENT-REQUIRED` / `PAYMENT-SIGNATURE` /
`PAYMENT-RESPONSE` headers. Its first scheme is **`exact`** (EIP-3009
`transferWithAuthorization`).

mandatehub does **not** re-implement on-chain settlement — x402 already solves that. Instead
mandatehub is the **mandate + proof layer** that sits inside a facilitator: it decides
*whether an autonomous agent is allowed to make this payment* (budget / policy / session-key /
window / replay), and it *proves* — offline — that every settled payment stayed within the
mandate. That is exactly the piece x402 leaves to the resource server.

## What ships today (`mandatehub.x402`)

A facilitator that mirrors x402's shape, with settlement pluggable:

- **`Facilitator.verify(requirements, payload)`** → a `VerifyResponse` computed by a
  side-effect-free mandate pre-authorization (`preauthorize`) — the ledger is never touched.
- **`Facilitator.settle(requirements, payload)`** → a `SettleResponse` that runs the payment
  through the mandate (posting the double-entry transfer) and returns a `ProofOfMandate`.
- **`PaymentRequirements` / `PaymentPayload`** using x402's vocabulary (`scheme`, `network`,
  `maxAmountRequired`, `payTo`, `asset`, `extra`), plus a mandatehub `extra` carrying the
  `mandateId` and `purpose`.
- **`serve_once` + the `PAYMENT-REQUIRED` / `PAYMENT-SIGNATURE` / `PAYMENT-RESPONSE` headers**
  — a stdlib-only HTTP 402 request cycle. [`examples/x402_facilitator.py`](../examples/x402_facilitator.py)
  runs it over a real localhost server: `402 → pay → 200 + proof`, and a replayed payment is
  rejected by the mandate over HTTP.

Settlement today uses the self-contained **`LedgerSettlementAdapter`** — it posts to the
mandatehub ledger, **no real money**. It is the mock/testnet substitute for on-chain settle.

## The seam: `SettlementAdapter`

```python
class SettlementAdapter(Protocol):
    def settle(self, engine, requirements, payload, *, at) -> tuple[bool, str, str]:
        """(success, tx_id, reason). Where the payment actually moves value."""
```

Swapping this is the *only* change needed to move real value: an adapter that forwards
`(paymentPayload, paymentRequirements)` to a real x402 facilitator (Coinbase CDP on Base) and
returns the on-chain tx hash — while mandatehub keeps doing the mandate gating + proofs around
it.

## Roadmap — "actually running", modeled on x402

x402 shipped as an open monorepo (a Python SDK under `./python/x402`, `docs/`, `specs/`,
`examples/`) with the `exact` scheme first and more `(scheme, network)` pairs added over time.
mandatehub follows the same shape:

- **Phase 1 — HTTP 402 facilitator (done).** `mandatehub.x402` with the `exact` scheme, the
  header protocol, mandate-gated verify/settle, and a live HTTP example. Settlement is the
  ledger adapter (no real money). Fully tested.
- **Phase 2 — real x402 v1 client (built; tested against stubs).** `mandatehub.x402` now
  ships a `RemoteFacilitatorAdapter` that speaks the real x402 **v1** facilitator protocol
  (`/verify`, `/settle`) over `urllib`, plus an `ExactEvmPayloadBuilder` that constructs the
  `exact`-scheme EIP-3009 + EIP-712 payment payload signed by a pluggable `Signer`. Verified
  against an in-process stub facilitator + `StubSigner` (no network, no keys). Running it
  against **real Base Sepolia** is the operator's last mile (facilitator URL, agent key,
  testnet USDC) — see [§ Phase 2 details](#phase-2-real-x402-v1-client) below.
- **Phase 3 — best-execution scheme.** Expose mandatehub's solver auction + surplus recapture
  (③) as an x402 scheme (e.g. `best-exec` alongside `exact`), returning the price-improvement
  surplus to the user — a value-add facilitator.
- **Phase 4 — mainnet hardening.** Durable storage (Postgres via the `LedgerStorage`
  protocol), key management, auth/rate-limiting, monitoring, a security review, signed
  releases, and a PyPI publish. Only then is real-money mainnet operation appropriate.

## Phase 2: real x402 v1 client

`mandatehub.x402` now includes a faithful **x402 v1** client for the `exact` (EVM/EIP-3009)
scheme — the format the live Coinbase CDP facilitator on Base speaks today. Everything below
is verified against an in-process stub facilitator + a `StubSigner`, so the wire format,
codec, request/response wiring, EIP-712 assembly, and every security guard are tested
**without any network or keys**.

Pieces (`mandatehub/x402/`, stdlib-only except the optional signer):

- **`wire.py`** — v1 wire dataclasses (`X402PaymentRequirements`, `X402PaymentPayload`,
  `EIP3009Authorization`, `FacilitatorVerifyResult`, `FacilitatorSettleResult`) with exact
  camelCase / `exclude_none` serialization and standard-padded base64 header codecs
  (`X-PAYMENT` / `X-PAYMENT-RESPONSE`).
- **`eip712.py`** — the `TransferWithAuthorization` typed-data assembly and the Base Sepolia
  constants (chain id `84532`, USDC `0x036CbD…CF7e`, EIP-712 domain `{name:"USDC",
  version:"2"}`; the domain separator was recomputed on-chain to a byte-exact match).
- **`signer.py`** — a `Signer` protocol with `NullSigner` / `StubSigner` (stdlib).
- **`exact_evm.py`** — `ExactEvmPayloadBuilder`: fresh 32-byte CSPRNG nonce, `value ==
  maxAmountRequired`, `to == payTo`, `validBefore = now + min(maxTimeoutSeconds, mandate_ttl)`.
- **`remote.py`** — `RemoteFacilitatorAdapter`: `https`-only (localhost exception), refuses
  cross-host redirects on `/verify`+`/settle`, never puts secrets in errors, raises on
  non-2xx / malformed JSON / network error (never treats failure as success), tolerant of
  unknown/`error`-vs-`errorReason` response keys.

Real EVM signing lives in **`mandatehub/signers/eth_account_signer.py`** behind the optional
`[evm]` extra (`pip install 'mandatehub[evm]'` → `eth-account`); the core never imports it.

### Operator's last mile (env / config)

To point at real Base Sepolia, supply (never hardcode):

| env var | purpose |
| ------- | ------- |
| `MANDATEHUB_FACILITATOR_URL` | facilitator base URL (`https://` only; `/verify`,`/settle` appended) |
| `MANDATEHUB_AGENT_PRIVATE_KEY` | EVM signer key (via keystore/KMS ideally) — consumed only by `EthAccountSigner`, never logged |
| `MANDATEHUB_NETWORK` | network slug (default `base-sepolia`) |
| `MANDATEHUB_CDP_API_KEY_*` | CDP auth, injected via the adapter's `header_hook` (CDP header spec still to confirm) |

Safe Base Sepolia defaults (chain id, USDC address, decimals, EIP-712 name/version) are
built in but overridable, and the builder still reads `asset`/`extra` from the runtime
requirements so other tokens/networks work.

### Still gated before mainnet

The v1 assumption should be confirmed with one live smoke test against the real CDP
facilitator (behind a `MANDATEHUB_LIVE_TEST=1` flag); the CDP auth-header format is
unconfirmed; and grant-side mitigations (server-derived requirements, atomic nonce
check-and-lock, deliver-only-after-`success:true`, fail-closed on `429`) belong to the
mandate-gate layer that wraps the adapter — see the security section of the internal spec.

## Honest status

Phase 1 is a **runnable protocol demo**, not a production payment service. It runs the x402
HTTP flow end-to-end and proves the mandate accounting, but it moves no real money and has no
persistence, auth, key management, or on-chain settlement yet. Phases 2–4 are where "actually
running with real value" is earned — each gated on decisions (network, custody, hosting) that
belong to the operator.
